Dispatching Fields

In this appendix, you will find brief definitions of each of the Dispatching fields, instructions on how to change or add information to them, and where in RouteMatch the information for that field was originally entered. If you are using Trip Mode, several of these fields will be split into separate pickup (PU) and drop-off (DO) fields.

 

If a field is automatically populated with data from a mobile device, any data that was entered manually in this field will be overwritten by the mobile device data. Therefore, if your agency uses mobile devices, we do not recommend editing these fields in the Dispatching module. They can be edited in the Verification module, if appropriate.

COP_Proposed Scheduled Status

This field indicates if any schedule violations existed for the trip in the proposed schedule.

This field cannot be edited. This field has four possible values: 0 = No schedule violations existed, 1 = Manual schedule violation existed, not changed by RSE, 2 = Based on the RSE semiclustering setting that allows this, RSE broke convergence or drift constraints for clustered trip, 4 = Forced clustering of trips was enabled in RSE settings and resulted in violation of one or more constraints for this trip.

COP_Proposed Validity Indicator

This field shows whether or not a new schedule has been proposed, the validity of the proposed schedule if one has been proposed, or whether the trip has been rescheduled.

This field cannot be edited. This field has four possible values: 1) None: there is no proposed scheduling change for this trip, 2) Optimized: there is a proposed change and it appears to still be valid, 3) Suspect: there were changes to the trip but NOT to the scheduled vehicle or scheduled pickup/dropoff times, 4) Rescheduled: there were changes to the trip that included either changing the scheduled vehicle or scheduled pickup/dropoff times.
